# RainbowChat 服务端：单机、无消息漫游 — 前端对接说明

本文档面向**仅单设备登录、不在多端间同步消息、不使用服务端增量漫游（SyncKey / user_seq）**的客户端实现。数据模型与接口以当前仓库 `javayuanma` 代码为准。

---

## 1. 架构原则（必读）

| 能力 | 单机无漫游下的约定 |
|------|-------------------|
| **会话与历史正文** | **以客户端本地数据库为准**。服务端 `missu_user_msgs_collect` 主要用于运营/审计/Web 等；移动端可不依赖「全量云端历史」接口。 |
| **断线期间未送达的 IM 消息** | 进入服务端离线表 `missu_offline_history`，上线后通过 HTTP **1008-4-8** 分批拉取并删除（服务端每批最多 500 条）。 |
| **实时聊天与信令** | 使用 **IM 长连接（MobileIMSDK）**：好友单聊、陌生人、群聊、部分系统通知等均走 `Protocal` + `typeu`。 |
| **HTTP** | 负责登录、鉴权、好友/群资料、离线补偿、文件上传等；与 IM 配合使用。 |

---

## 2. HTTP 传输约定

### 2.1 入口

- **主入口**：`POST {contextPath}/rest_post`（见 `web.xml` → `MyControllerJSON`）。
- **请求体**：EVA.EPC 框架约定为 **`DataFromClient` 的 Java 序列化**（`ObjectInputStream` / `ObjectOutputStream`），**不是**纯 JSON Body。官方 RainbowChat 客户端与此一致。
- **响应体**：封装为 **`DataFromServer`**（含 `success`、`returnValue`、`code` 等）；业务 JSON 一般在 `returnValue` 中（登录等部分方法直接返回字符串，以实际客户端解析为准）。

若你们用 H5 / 跨语言栈，需单独做**网关层**把 JSON 转为与 `DataFromClient` 等价的序列化协议，或改用 `MyControllerJSONP`（`/rest_get`，JSONP，适用场景有限）。

### 2.2 `DataFromClient` 关键字段

| 字段 | 含义 |
|------|------|
| `processorId` | 处理器 ID（如 `1008` 业务逻辑、`1016` 群组、`1017` 登录）。 |
| `jobDispatchId` | 作业域（如好友 `2`、消息 `4`、群查询 `25`）。 |
| `actionId` | 具体动作（如 `7`、`8`… 对应各接口）。 |
| `newData` / `oldData` | 多为 **JSON 字符串**（依接口而定）。 |
| `token` | 登录成功后服务端下发的 Token；**若开启 HTTP Token 校验**，除登录外每次请求需携带。 |
| `device`（继承自 `Device`） | 平台类型等，与推送、部分逻辑相关，需与官方约定一致。 |

**接口编号**与代码一致：`{processorId}-{jobDispatchId}-{actionId}`，例如 `1008-4-8` 表示处理器 `1008`、作业 `4`、动作 `8`。

## 4. 推荐对接流程（顺序）

下列顺序在**单机、无漫游**下最稳妥：先 HTTP 拿关系与离线，再挂 IM，最后用 IM 收实时。

```
1017 登录
  → 持久化 user_uid、token、device 等
  →（若 TOKEN_FOR_HTTP_VERIFY_ENABLED=true）后续所有业务 HTTP 带 token

HTTP 并行/依次：
  → 1008-2-7  好友列表
  → 1016-25-7 我的群列表
  → 按需 1016-25-8 / 1016-25-9 群信息、群成员
  → 1008-4-7  离线加好友请求（如有）
  → 1008-4-8  循环直到返回空数组：离线聊天消息

IM：
  → 使用 user_uid + token 连接（见第 5 节）
  → 上线后实时消息仅走 IM
  → 好友关系变更：除 **1008-2-7** 外，须处理 **MT10**（同意/免验证后下发对方资料）、**MT13**（被加好友通知）以实时刷新本地好友表（见 §8.1.1）

可选：
  → 1008-4-9  未处理加好友请求数 + 最近一条请求摘要（角标/首屏，详见 §8.2.1）
  → 1008-4-24 / 1008-4-25  已读水位上报与查询（详见 §8.4）
  → 注销：processorId = -2，newData 为 LogoutInfo 的 JSON
```

**不建议在「单机无漫游」方案中接入**（避免与多端/漫游耦合）：

- **1008-4-27 / 1008-4-28**：SyncKey 增量与确认（典型漫游/多端同步）。
- **1008-4-30**：状态同步日志（多端）。
- **1008-4-38**：`USER_SEQ_SYNC_ENABLED` 方案 B 事件流。
- 不必实现「本机发出消息依赖服务端 Echo 多端同步」的完整链路；单端以**本地插入 + IM 收对方**为主即可。

## 5. 登录 **1017**

| 项 | 值 |
|----|-----|
| `processorId` | `1017` |
| `newData` | `LoginInfo2` 的 **JSON 字符串** |

`LoginInfo2` 常用字段：

| 字段 | 说明 |
|------|------|
| `loginType` | `"0"` 密码登录，`"1"` 短信验证码登录 |
| `loginName` | 密码登录时为账号/邮箱；短信登录时为手机号 |
| `loginPsw` | 明文密码或短信验证码 |
| `loginPswCrypt` | 自动登录等场景密文（按现有 App 约定） |
| `appKey` | 须与服务端 `base_conf.properties` 中 `APPKEY_GRANTED` 一致（若开启 `APPKEY_VERIFY_ENABLED`） |
| `clientVersion` | 客户端版本号 |
| `deviceInfo` | 设备说明（可选） |
| `deviceID` | iOS 时为 APNs device token 等（推送用） |
| `hardwareId` | 稳定设备 ID（新设备校验等，见代码注释） |
| `deviceVerifyCode` | 新设备二次验证短信码 |

**成功返回**（`returnValue` 内 JSON）：含 `authed_info`（字符串化的 `UserEntity`），解析后得到：

- **`user_uid`**：全系统用户唯一 ID，**IM 登录用户名**使用此字段。
- **`token`**：与 `user_uid` 配对；**IM 连接密码位**使用此 token（见 `UserEntity` 注释：loginUserId + loginToken）。

**失败**：含 `authed_code`、可能的 `masked_phone`、`freeze_remain_seconds` 等（见 `MyControllerJSON.doPROCESSOR_LOGIN_4ALL_V2`）。

## 6. IM 长连接（MobileIMSDK）

### 6.1 连接参数

| 参数 | 来源 |
|------|------|
| 用户 ID | `UserEntity.user_uid` |
| Token | `UserEntity.token` |
| 服务器 IP/域名 | **客户端配置**，须与部署环境一致（代码中 `BaseConf`：`IM_SERVER_TCP_LISTENNING_PORT`、`IM_SERVER_UDP_LISTENNING_PORT` 等） |
| TCP/UDP/WebSocket | 与 `base_conf.properties` 中 `IM_SERVER_TCP_SUPPORT`、`IM_SERVER_UDP_SUPPORT`、`IM_SERVER_WEBSOCKET_SUPPORT` 一致 |

### 6.2 鉴权开关

- `TOKEN_FOR_SOCKET_VERIFY_ENABLED`：为 `true` 时，IM 登录必须带合法 `token`（`TokenManager.verifySocketToken`）。

### 6.3 常见业务 `typeu`（`UserProtocalsType`，与官方 RainbowChat / MobileIMSDK 一致）

| 场景 | 方向 | 说明 |
|------|------|------|
| 好友单聊消息 | 客户端 → 客户端，**MT03** | 服务端在发送成功回调中落库、送达通知等 |
| 陌生人/临时会话 | **MT42** → `to=0` 服务端 | 服务端转 **MT43** 给接收方 |
| 群聊 | **MT44** → `to=0` | 服务端扇出 **MT45** 等；大群 `group_mode=2` 见下节 |

具体消息体为 JSON（`MsgBody4Friend`、`MsgBody4Guest`、`MsgBody4Group` 等），须与现有协议字段一致。

## 7. 大群读扩散（与漫游无关，但与群类型有关）

当群 **`group_mode == 2`**（大群读扩散）时，成员不依赖传统「每条群消息扇出到离线表」的模式，需通过 HTTP 按序号拉取：

| 接口 | 说明 |
|------|------|
| **1016-25-25** | 按 `seq` 拉取群消息（见 `GroupChatProcessor` 注释） |

客户端进入该群会话时，应结合群信息里的 `group_mode` 决定走 IM 实时 + **1016-25-25** 补历史，还是普通群仅 IM。

## 8. 核心业务 HTTP 清单（单机无漫游最小集）

### 8.1 好友 **1008-2-***

| 接口编号 | `jobDispatchId` | `actionId` | `newData` | 说明 |
|----------|-----------------|------------|-----------|------|
| **1008-2-7** | 2 | 7 | 当前用户 **uid 字符串** | 好友列表（含在线状态等） |

拉黑、备注等按需使用 **1008-2-27**、**1008-2-8** 等（见 `LogicProcessor2.maintainRosterMgrJobDispatcher`）。

### 8.1.1 加好友「免验证」时如何实时更新好友列表

当被加方在隐私里设置 **`require_verification = 0`**（不需要验证、自动同意）时，服务端在收到 A 的加好友请求（IM **MT05** → `ChatLogicManager.processMT05_...`）后会**直接写入双向 `missu_roster`**，并通过 **IM 主动下发两条「好友信息」**，客户端应**监听 IM、解析 JSON、写入本地好友表**，而**不必**等用户再点「同意」或再调一次 **1008-2-7** 才能看到对方（当然仍可用 2-7 做全量校正）。

| 接收方 | IM `typeu`（常量名） | `dataContent` | 客户端建议动作 |
|--------|----------------------|---------------|----------------|
| **发起方 A** | **MT10**（`MT10_OF_PROCESS_ADD$FRIEND$REQ_FRIEND$INFO$SERVER$TO$CLIENT`） | 被加方 **B** 的 `UserEntity` JSON | 与「对方已同意加好友」同一条链路：把 **B** 插入/更新本地好友列表 |
| **被加方 B** | **MT13**（`MT13_OF_BE_ADDED_AS_FRIEND_NOTIFY_SERVER$TO$B`） | 发起方 **A** 的 `UserEntity` JSON | 把 **A** 插入/更新本地好友列表（免验证下由服务端代发，无需 B 手动确认） |

说明（与代码一致）：

- 两条消息均由服务端发起（`MessageHelper` 中 `from` 为服务端约定值，带 QoS），**走 IM 实时通道**。
- 若 **B 当时不在线**，**MT13** 会走发送失败回调并进入 **`OfflineMessageManager`**（与桥接/离线逻辑一致），B **重连后**会在 **1008-4-8** 的离线记录中收到同类通知（`typeu` 含 MT13 分支），客户端解析后同样应合并进好友列表。
- **兜底**：若客户端暂未实现 MT10/MT13 处理，可在收到任意「好友关系可能已变」的事件后调用 **1008-2-7** 全量拉取好友列表（非实时、但数据一致）。

### 8.2 离线消息与好友请求 **1008-4-***

| 接口编号 | `jobDispatchId` | `actionId` | `newData` | 说明 |
|----------|-----------------|------------|-----------|------|
| **1008-4-7** | 4 | 7 | 本地用户 uid 字符串 | 离线时收到的**加好友请求**列表 |
| **1008-4-8** | 4 | 8 | **JSON 对象字符串**（见下表） | **离线聊天消息**；每批最多 500 条，拉取后服务端删除本批 |
| **1008-4-9** | 4 | 9 | 当前用户 **uid 字符串**（与 4-7 相同，直接传 `user_uid`） | 未处理加好友请求**数量**及**最近一条**请求摘要（见 §8.2.1） |

**1008-4-8** 请求 JSON 字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_uid` | string | 离线消息**接收者**（当前登录用户） |
| `from_user_uid` | string，可 null | 只拉某发送方时填写；null 表示拉取所有人发给「我」的离线消息 |
| `include_fp` | string | `"1"` 返回指纹 `fp`；`"0"` 或不传可按需 |

响应为 `OfflineMsgDTO` 数组的 JSON；**循环调用直到返回 `[]`**，避免单次超 500 条未拉完。

### 8.2.1 **1008-4-9** 未处理加好友请求（角标 / 首屏）

| 项 | 说明 |
|----|------|
| `processorId` | `1008` |
| `jobDispatchId` | `4`（`LOGIC_MESSAGES`） |
| `actionId` | `9`（`ACTION_APPEND3`） |
| `newData` | **字符串**：当前登录用户的 `user_uid`（与 **1008-4-7** 一致，不是 JSON 包一层） |

**成功时** `returnValue` 为 JSON 对象，字段如下：

| 字段 | 类型 | 说明 |
|------|------|------|
| `req_count` | string | 未处理的加好友请求**总数**（数字以字符串形式返回，如 `"3"`） |
| `latest_req_obj` | string，可选 | **最近一条**加好友请求的 JSON 文本（内容结构与离线列表里单条一致，便于首屏展示）；若无待处理请求则可能不出现该字段 |

**用途**：在 **1008-4-7** 全量拉列表之外，用于 Tab 角标、通知摘要等；统计口径与 4-7 相同（排除系统内置排除账号 `FRIEND_REQ_EXCLUDE_UID_LIST` 中的请求方）。

**与 1008-4-7 的关系**：4-7 返回的是待处理请求**列表**（用于逐条处理）；4-9 返回**计数 + 最新一条摘要**，不删除好友请求表数据。

### 8.3 群查询 **1016-25-***

| 接口编号 | `processorId` | `jobDispatchId` | `actionId` | 说明 |
|----------|---------------|-----------------|------------|------|
| **1016-25-7** | 1016 | 25 | 7 | 我的群列表 |
| **1016-25-8** | 1016 | 25 | 8 | 群基本信息 |
| **1016-25-9** | 1016 | 25 | 9 | 群成员列表（大群可能分页，单次上限见服务端实现） |
| **1016-25-25** | 1016 | 25 | 25 | 大群读扩散按 seq 拉消息 |

建群、邀请、退群、禁言等走 **1016-24-***（`LOGIC_GROUP_BASE_MGR` = 24），按产品需要在 `GroupChatProcessor` 中查具体 action。

### 8.4 已读水位（可选）：**1008-4-24** 上报、**1008-4-25** 查询

单机无漫游时，仍可用已读水位做「单聊/陌生人/群内，对方已读到什么时间戳」的展示；不要求跨设备一致时可只在本机结合 **1008-4-25** 拉一次对方水位用于 UI。

#### **1008-4-24** 上报已读（HTTP，等价于 IM **MT64**）

| 项 | 说明 |
|----|------|
| `processorId` | `1008` |
| `jobDispatchId` | `4` |
| `actionId` | `24`（`ACTION_APPEND6`） |
| `newData` | **JSON 对象字符串**，字段见下表 |

**请求 JSON 字段**（`LogicProcessor2` → `applyReadReceiptWaterline`）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `luid` | string | 当前用户 uid（读消息的人） |
| `partner_id` | string | 会话对象：单聊/陌生人为对方 uid；群聊为 **群 id** |
| `chat_type` | string | `"0"` 好友单聊、`"1"` 陌生人、`"2"` 群聊 |
| `last_read_time2` | string | 已读到的**最新消息**时间戳（与消息体中的 `msg_time2` / 展示用毫秒一致，服务端按字符串写入水位） |

**返回**：`"1"` 表示写入成功，`"0"` 表示失败。

**服务端行为摘要**（实现细节以代码为准）：写入/更新 `missu_msg_read_receipt`；若对方在线，可能通过 **IM** 推送已读通知；若开启多端状态同步，可能写状态日志（单机方案可忽略对端设备表现）。

**IM 等价路径**：`typeu = MT64`，`to = 0`，`dataContent` 为与上表相同字段的 JSON（见 `ChatServerEventListener`、`LogicProcessor2.applyReadReceiptFromImSocket`）。任选 HTTP 或 IM 一种上报即可，避免双写。

#### **1008-4-25** 查询对方已读水位

| 项 | 说明 |
|----|------|
| `processorId` | `1008` |
| `jobDispatchId` | `4` |
| `actionId` | `25`（`ACTION_APPEND7`） |
| `newData` | **JSON 对象字符串**，字段：`luid`（我的 uid）、`partner_id`（对方 uid 或群 id）、`chat_type`（同上 `0/1/2`） |

**返回**：JSON，`last_read_time2` 为字符串；若无记录则为 `"0"`。

**语义**：查询 **partner_id** 在与 **luid** 的会话中已读到的时间戳（即「对方读我到哪」用于双勾等展示时，注意与产品文案一致）。

## 9. Token 与安全（`base_conf.properties`）

| 配置项 | 为 `true` 时前端行为 |
|--------|---------------------|
| `TOKEN_FOR_HTTP_VERIFY_ENABLED` | 除登录、注销等白名单外，**每次** `DataFromClient.token` 必传且有效 |
| `TOKEN_FOR_SOCKET_VERIFY_ENABLED` | IM 登录 **token** 必为登录接口下发的值 |
| `APPKEY_VERIFY_ENABLED` | `LoginInfo2.appKey` 必须等于 `APPKEY_GRANTED` |

生产环境建议开启 HTTP + Socket Token。

---

## 10. 本地消息与去重建议

1. **在线消息**：IM 收到即展示并写入本地；`Protocal` 带 **指纹 `fp`**，应用层应按 `fp` 去重（服务端注释：QoS 下可能重复回调）。  
2. **离线消息**：**1008-4-8** 拉到的记录写入本地前同样建议按服务端返回的指纹字段去重。  
3. **自己发送的消息**：单机方案下可在发送成功（客户端 SDK 回调）即写本地，不依赖跨端 Echo。

---

## 11. 富媒体与文件

图片、语音、大文件等走独立 **Servlet**（如 `MsgImageUploader`、`BigFileUploader` 等，见 `web.xml`）及处理器 **1011**（`PROCESSOR_UPLOAD_BINARY`）。消息体内一般携带文件名或 URL，与官方客户端字段保持一致即可。
