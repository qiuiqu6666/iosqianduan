当前 checkout 的核心事实只有两条：

- 这是一个以 `RainbowChat4i` 为核心的 iOS 原生即时通讯客户端工程。
- 仓库里保留了服务端接口、部署、协议的痕迹和文档，但没有实际的 `RainbowChatServer/` 源码目录。

## 文档边界

### 这套 Wiki 覆盖什么

- 仓库结构和模块职责
- iOS 客户端启动、登录、导航、页面分层
- HTTP、IM、Provider、SQLite 的调用关系
- 会话列表、消息、已读、离线同步、SyncKey 增量同步
- Xcode 构建、签名、运行、发版和常见排障入口
- 服务端在当前仓库中的缺失边界，以及客户端对外部服务的依赖

### 这套 Wiki 不覆盖什么

- 不臆造不存在的服务端实现细节
- 不把旧文档中的 `RainbowChatServer` 当成当前 checkout 里的可维护代码
- 不逐个解释每一张图片资源和每一个三方库的内部实现

## 先记住三件事

1. 这是原生 iOS IM 工程，不是 Web 项目。
2. 这是“HTTP 业务接口 + IM 长连接 + 本地 SQLite”三条线一起工作的客户端。
3. 当前仓库里没有服务端源码，任何服务端问题都只能从接口、配置和客户端调用点倒推。

## 推荐维护策略

- 新增功能时，优先更新对应模块文档，而不是只改首页。
- 只要改到启动链路、服务地址、同步协议、未读逻辑，就同步更新 `01`、`03`、`04`、`05`。
- 如果后续真的补回服务端源码，再新增服务端专属 Wiki，不要直接把旧推测写回当前文档。
