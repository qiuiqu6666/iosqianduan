# 服务端边界与外部依赖

## 1. 先把边界说清楚

当前仓库里没有 `RainbowChatServer/` 源码目录。

所以这套 Wiki 不会把服务端当成“当前可读、可改、可编译”的仓库内模块。

更准确地说：

- 仓库里有大量服务端存在过的证据
- 但没有服务端的实际落盘代码
- 客户端仍然强依赖外部服务端能力

## 2. 服务端存在过的证据有哪些

| 证据 | 说明 |
| --- | --- |
| `Default.h` 中的 HTTP / IM 地址 | 客户端需要真实后端 |
| `HttpRestHelper` 大量接口 | 说明服务端业务面很广 |
| `MT47_MT50_MT51_前端对接文档.md` | 存在明确的协议约定 |
| `单机无漫游前端对接.md` | 说明后端能力存在不同运行模式 |
| 旧版 Wiki 文档 | 说明历史上曾配套服务端材料 |

## 3. 从客户端视角反推，服务端至少负责什么

### 3.1 HTTP 业务层

客户端已经明确依赖服务端提供下面这些 HTTP 能力：

- 登录认证
- 好友和群相关接口
- 会话列表接口
- 离线消息接口
- SyncKey 增量同步接口
- 已读状态接口
- 钱包、红包、转账接口
- 文件上传下载接口
- 音视频相关辅助接口

### 3.2 IM 实时层

客户端还依赖一层独立的 IM 通道，至少负责：

- Socket 登录
- 实时消息转发
- ACK / QoS / 送达回执
- 已读回执
- 撤回和系统通知
- 断线重连后的状态恢复

### 3.3 文件与媒体层

从客户端接口命名可以看出，服务端还承担：

- 图片上传
- 语音上传
- 头像上传
- 大文件上传下载
- 短视频上传下载

## 4. 对前端 / 后端 / 运维分别意味着什么

### 4.1 对客户端开发

你不能只看本地代码来修所有问题。

以下场景天然要联动服务端：

- 登录失败
- 会话列表未读不准
- SyncKey 卡住
- 文件上传失败
- 已读对不上
- 钱包数据不对

### 4.2 对后端开发

如果后端同学接手的是另外一个仓库，需要优先和客户端对齐这几件事：

- 接口路径和参数格式
- IM 消息体结构
- `syncKey` / `state_ops` 语义
- 会话列表 `26-7` 的数据格式
- 已读和送达事件语义

### 4.3 对运维 / 交付

你需要准备的不是一个服务，而是一组服务：

- HTTP 业务接口
- IM 长连接端口
- 上传下载服务
- 推送能力
- Agora 相关支持
- 地图 key

## 5. 当前仓库里的“后端阅读入口”

## 6. 服务端缺失带来的实际影响

### 6.1 对代码理解的影响

- 你能看清客户端怎么调接口。
- 你能看清客户端期待什么返回。
- 你看不见服务端为什么这么返回。

### 6.2 对开发的影响

- 客户端 UI 开发还能继续做。
- 很多接口联调、未读修复、多端一致性问题，必须有真实后端才能闭环验证。

### 6.3 对 Wiki 的影响

所以当前 Wiki 采用的原则是：

- 只写客户端可证实的事实
- 服务端只写边界和依赖，不写伪实现

## 8. 一句话总结

当前仓库的真实边界是：客户端代码齐全，服务端只剩接口痕迹和历史文档，任何后端问题都要按“外部依赖”来理解和协作。
